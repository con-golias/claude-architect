## Memory Safety Weaknesses

### CWE-787: Out-of-bounds Write

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 787                                        |
| Name            | Out-of-bounds Write                        |
| 2025 Rank       | 1                                          |
| 2024 Rank       | 1                                          |
| Change          | Unchanged                                  |
| Affected Languages | C, C++, Assembly                        |
| CVSS Range      | 7.0 - 10.0 (High to Critical)             |

#### Description

An out-of-bounds write occurs when software writes data past the end (or before the beginning) of a buffer. This can corrupt adjacent memory, overwrite control data (function pointers, return addresses), and lead to arbitrary code execution. This has been the #1 most dangerous weakness for multiple consecutive years.

#### Vulnerable Code -- C

```c
// VULNERABLE: Stack buffer overflow
void process_input(const char *input) {
    char buffer[64];
    strcpy(buffer, input);  // No bounds checking -- overflow if input > 63 chars
    printf("Processed: %s\n", buffer);
}

// VULNERABLE: Heap buffer overflow
void resize_array(int *arr, int old_size, int new_size) {
    int *new_arr = malloc(new_size * sizeof(int));
    // No check: if old_size > new_size, writes out of bounds
    memcpy(new_arr, arr, old_size * sizeof(int));
    free(arr);
}
```

#### Secure Code -- C

```c
// SECURE: Bounds-checked string copy
void process_input(const char *input) {
    char buffer[64];
    size_t input_len = strlen(input);
    if (input_len >= sizeof(buffer)) {
        fprintf(stderr, "Input too long\n");
        return;
    }
    strncpy(buffer, input, sizeof(buffer) - 1);
    buffer[sizeof(buffer) - 1] = '\0';
    printf("Processed: %s\n", buffer);
}

// SECURE: Validated memory copy
void resize_array(int *arr, int old_size, int new_size) {
    if (new_size <= 0 || old_size < 0) return;
    int *new_arr = malloc(new_size * sizeof(int));
    if (!new_arr) return;
    int copy_size = (old_size < new_size) ? old_size : new_size;
    memcpy(new_arr, arr, copy_size * sizeof(int));
    free(arr);
}
```

#### Secure Code -- Rust (Memory-Safe Alternative)

```rust
// SECURE: Rust prevents out-of-bounds writes at compile time
fn process_input(input: &str) {
    let mut buffer = [0u8; 64];
    let bytes = input.as_bytes();
    let copy_len = bytes.len().min(buffer.len());
    buffer[..copy_len].copy_from_slice(&bytes[..copy_len]);
    // Bounds are checked at compile time and runtime
}

fn resize_array(arr: &[i32], new_size: usize) -> Vec<i32> {
    let mut new_arr = Vec::with_capacity(new_size);
    let copy_len = arr.len().min(new_size);
    new_arr.extend_from_slice(&arr[..copy_len]);
    new_arr.resize(new_size, 0);
    new_arr
}
```

#### Prevention

- [ ] Use memory-safe languages (Rust, Go, Java, Python) where possible
- [ ] Use bounds-checked buffer functions (strncpy, snprintf instead of strcpy, sprintf)
- [ ] Enable compiler protections (stack canaries, ASLR, DEP/NX)
- [ ] Use static analysis tools (Coverity, CodeQL, clang-tidy)
- [ ] Use AddressSanitizer (ASan) during development and testing

### CWE-125: Out-of-bounds Read

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 125                                        |
| Name            | Out-of-bounds Read                         |
| 2025 Rank       | 3                                          |
| 2024 Rank       | 4                                          |
| Change          | Up 1                                       |
| Affected Languages | C, C++                                  |
| CVSS Range      | 4.0 - 8.0 (Medium to High)                |

#### Description

An out-of-bounds read occurs when software reads data from before the beginning or past the end of a buffer. This can leak sensitive data from memory (including cryptographic keys and passwords) or cause crashes. The Heartbleed vulnerability (CVE-2014-0160) was a famous out-of-bounds read.

#### Vulnerable Code -- C

```c
// VULNERABLE: Out-of-bounds read
char* get_value(const char *data, int index) {
    // No bounds check -- can read beyond buffer
    return &data[index];
}

// VULNERABLE: Off-by-one read
void print_last_element(int *arr, int size) {
    printf("Last: %d\n", arr[size]); // Should be arr[size - 1]
}
```

#### Secure Code -- C

```c
// SECURE: Bounds-checked read
char* get_value(const char *data, int data_len, int index) {
    if (index < 0 || index >= data_len) {
        return NULL;  // Out of bounds
    }
    return (char*)&data[index];
}

// SECURE: Correct index
void print_last_element(int *arr, int size) {
    if (size <= 0) return;
    printf("Last: %d\n", arr[size - 1]);
}
```

#### Prevention

- [ ] Validate all array indices and buffer offsets before access
- [ ] Use memory-safe languages or safe abstractions
- [ ] Enable AddressSanitizer during testing
- [ ] Use static analysis to detect potential out-of-bounds reads

### CWE-416: Use After Free

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 416                                        |
| Name            | Use After Free                             |
| 2025 Rank       | 5                                          |
| 2024 Rank       | 7                                          |
| Change          | Up 2                                       |
| Affected Languages | C, C++                                  |
| CVSS Range      | 7.0 - 10.0 (High to Critical)             |

#### Description

Use After Free occurs when a program continues to reference memory after it has been freed. The freed memory may be reallocated for a different purpose, so writing to or reading from it can corrupt data or enable code execution. This is a primary vector for browser and kernel exploits.

#### Vulnerable Code -- C

```c
// VULNERABLE: Use after free
void process_data() {
    char *buffer = malloc(256);
    // ... use buffer ...
    free(buffer);

    // Later in the code...
    strcpy(buffer, "new data"); // USE AFTER FREE: buffer is freed
}

// VULNERABLE: Double free
void cleanup(char *ptr, int error) {
    free(ptr);
    if (error) {
        free(ptr); // DOUBLE FREE
    }
}
```

#### Secure Code -- C

```c
// SECURE: Null pointer after free
void process_data() {
    char *buffer = malloc(256);
    if (!buffer) return;
    // ... use buffer ...
    free(buffer);
    buffer = NULL; // Prevent use after free

    // Later use will segfault (detectable) instead of corrupt memory
    if (buffer != NULL) {
        strcpy(buffer, "new data");
    }
}

// SECURE: Track free state
void cleanup(char **ptr, int error) {
    if (*ptr != NULL) {
        free(*ptr);
        *ptr = NULL;
    }
}
```

#### Secure Code -- Rust

```rust
// SECURE: Rust ownership system prevents use-after-free at compile time
fn process_data() {
    let buffer = String::from("data");
    drop(buffer); // Explicit free
    // println!("{}", buffer);  // COMPILE ERROR: value moved
}
```

#### Prevention

- [ ] Set pointers to NULL after freeing
- [ ] Use smart pointers in C++ (unique_ptr, shared_ptr)
- [ ] Use memory-safe languages with garbage collection or ownership systems
- [ ] Use AddressSanitizer to detect use-after-free during testing
- [ ] Adopt RAII patterns in C++

### CWE-476: NULL Pointer Dereference

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 476                                        |
| Name            | NULL Pointer Dereference                   |
| 2025 Rank       | 7                                          |
| 2024 Rank       | 6                                          |
| Change          | Down 1                                     |
| Affected Languages | C, C++, Java, C#                        |
| CVSS Range      | 4.0 - 7.5 (Medium to High)                |

#### Description

A NULL pointer dereference occurs when a program attempts to use a pointer that is NULL (points to address 0). This typically causes a crash (denial of service) and in some environments can be exploited for code execution through NULL page mapping.

#### Vulnerable Code -- C

```c
// VULNERABLE: No null check
void process(struct User *user) {
    printf("Name: %s\n", user->name); // Crash if user is NULL
}

// VULNERABLE: Check after dereference (too late)
void handle(struct Config *cfg) {
    int value = cfg->timeout; // Dereference before check
    if (cfg == NULL) return;  // Too late -- already crashed
}
```

#### Secure Code -- C

```c
// SECURE: Check before dereference
void process(const struct User *user) {
    if (user == NULL) {
        fprintf(stderr, "Error: user is NULL\n");
        return;
    }
    printf("Name: %s\n", user->name);
}
```

#### Secure Code -- Go

```go
// SECURE: Go's nil checks are enforced idiomatically
func process(user *User) error {
    if user == nil {
        return fmt.Errorf("user is nil")
    }
    fmt.Printf("Name: %s\n", user.Name)
    return nil
}
```

#### Prevention

- [ ] Check pointers and references for NULL/nil before dereferencing
- [ ] Use Option/Optional types (Rust Option, Java Optional, TypeScript strict null checks)
- [ ] Enable compiler null safety features
- [ ] Use static analysis to detect potential null dereferences

### CWE-190: Integer Overflow or Wraparound

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 190                                        |
| Name            | Integer Overflow or Wraparound             |
| 2025 Rank       | 10                                         |
| 2024 Rank       | 8                                          |
| Change          | Down 2                                     |
| Affected Languages | C, C++, Java                            |
| CVSS Range      | 5.0 - 9.8 (Medium to Critical)            |

#### Description

An integer overflow occurs when an arithmetic operation produces a value that exceeds the maximum (or minimum) value for the integer type. In C/C++, this results in wraparound, which can lead to undersized buffer allocations, incorrect loop bounds, or logic errors.

#### Vulnerable Code -- C

```c
// VULNERABLE: Integer overflow in allocation size
void *allocate_array(int count, int element_size) {
    // If count * element_size overflows, a small buffer is allocated
    int total_size = count * element_size; // Can overflow
    return malloc(total_size);
}

// VULNERABLE: Size_t wraparound
void copy_data(const char *src, size_t offset, size_t length) {
    size_t end = offset + length; // Can wrap around to small value
    char *dest = malloc(length);
    memcpy(dest, src + offset, length);
}
```

#### Secure Code -- C

```c
// SECURE: Overflow-checked allocation
#include <stdint.h>

void *allocate_array(size_t count, size_t element_size) {
    // Check for multiplication overflow
    if (count > 0 && element_size > SIZE_MAX / count) {
        return NULL; // Would overflow
    }
    size_t total_size = count * element_size;
    return malloc(total_size);
}

// Or use calloc which checks internally
void *safe_allocate(size_t count, size_t element_size) {
    return calloc(count, element_size); // Checks for overflow internally
}
```

#### Secure Code -- Rust

```rust
// SECURE: Rust checks for integer overflow in debug mode
fn allocate_array(count: usize, element_size: usize) -> Option<Vec<u8>> {
    let total_size = count.checked_mul(element_size)?; // Returns None on overflow
    let mut vec = Vec::with_capacity(total_size);
    vec.resize(total_size, 0);
    Some(vec)
}
```

#### Prevention

- [ ] Use checked arithmetic operations that detect overflow
- [ ] Validate input values before arithmetic operations
- [ ] Use larger integer types or arbitrary-precision integers for critical calculations
- [ ] Use safe allocation functions (calloc instead of malloc with manual multiplication)
- [ ] Enable compiler warnings for integer truncation and conversion

## Injection Weaknesses

### CWE-79: Cross-site Scripting (XSS)

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 79                                         |
| Name            | Improper Neutralization of Input During Web Page Generation (XSS) |
| 2025 Rank       | 2                                          |
| 2024 Rank       | 2                                          |
| Change          | Unchanged                                  |
| Affected Languages | JavaScript, TypeScript, PHP, Java, Python |
| CVSS Range      | 4.0 - 9.0 (Medium to Critical)            |

#### Description

Cross-site Scripting occurs when an application includes untrusted data in a web page without proper validation or encoding. Attackers can execute scripts in victims' browsers, hijacking sessions, defacing websites, or redirecting users to malicious sites. There are three types: reflected, stored, and DOM-based XSS.

#### Vulnerable Code -- TypeScript

```typescript
// VULNERABLE: Reflected XSS
app.get('/search', (req, res) => {
  const query = req.query.q;
  res.send(`<h1>Results for: ${query}</h1>`); // No encoding
});

// VULNERABLE: DOM-based XSS
// document.getElementById('output').innerHTML = location.hash.substring(1);
```

#### Secure Code -- TypeScript

```typescript
// SECURE: Output encoding and CSP
import escapeHtml from 'escape-html';

app.get('/search', (req, res) => {
  const query = escapeHtml(req.query.q as string);
  res.send(`<h1>Results for: ${query}</h1>`);
});

// Or use a template engine with auto-escaping
app.get('/search', (req, res) => {
  res.render('search', { query: req.query.q }); // Auto-escaped by template
});

// Set CSP header
app.use((req, res, next) => {
  res.setHeader('Content-Security-Policy', "default-src 'self'; script-src 'self'");
  next();
});
```

#### Prevention

- [ ] Use template engines with auto-escaping (React, Angular, Jinja2)
- [ ] Apply context-appropriate output encoding (HTML, URL, JavaScript, CSS)
- [ ] Implement Content Security Policy (CSP) headers
- [ ] Use HttpOnly and Secure flags on session cookies
- [ ] Validate and sanitize all user input server-side

### CWE-89: SQL Injection

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 89                                         |
| Name            | Improper Neutralization of Special Elements in SQL Command |
| 2025 Rank       | 4                                          |
| 2024 Rank       | 3                                          |
| Change          | Down 1                                     |
| Affected Languages | All (where SQL is used)                 |
| CVSS Range      | 7.0 - 10.0 (High to Critical)             |

#### Description

SQL Injection occurs when untrusted data is sent to a SQL interpreter as part of a query. Attackers can read, modify, or delete database data, bypass authentication, and sometimes execute operating system commands on the database server.

#### Vulnerable Code -- Python

```python
# VULNERABLE: String concatenation in SQL
def get_user(username):
    query = f"SELECT * FROM users WHERE username = '{username}'"
    cursor.execute(query)
    return cursor.fetchone()
```

#### Secure Code -- Python

```python
# SECURE: Parameterized query
def get_user(username):
    cursor.execute("SELECT * FROM users WHERE username = %s", (username,))
    return cursor.fetchone()
```

#### Vulnerable Code -- Go

```go
// VULNERABLE: String formatting
func getUser(db *sql.DB, username string) (*User, error) {
    query := fmt.Sprintf("SELECT * FROM users WHERE username = '%s'", username)
    return db.QueryRow(query).Scan(...)
}
```

#### Secure Code -- Go

```go
// SECURE: Parameterized query
func getUser(db *sql.DB, username string) (*User, error) {
    var user User
    err := db.QueryRow("SELECT id, name, email FROM users WHERE username = $1", username).
        Scan(&user.ID, &user.Name, &user.Email)
    return &user, err
}
```

#### Prevention

- [ ] Use parameterized queries (prepared statements) exclusively
- [ ] Use ORM frameworks with built-in parameterization
- [ ] Apply input validation as a defense-in-depth measure
- [ ] Apply the principle of least privilege to database accounts
- [ ] Use stored procedures where appropriate

### CWE-78: OS Command Injection

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 78                                         |
| Name            | Improper Neutralization of Special Elements in OS Command |
| 2025 Rank       | 6                                          |
| 2024 Rank       | 5                                          |
| Change          | Down 1                                     |
| Affected Languages | All (where shell commands are invoked)  |
| CVSS Range      | 8.0 - 10.0 (High to Critical)             |

#### Description

OS Command Injection occurs when an application passes untrusted data to a system shell. Attackers can inject shell metacharacters to execute arbitrary commands on the host operating system, potentially gaining full server control.

#### Vulnerable Code -- Python

```python
# VULNERABLE: Shell command injection
import os

def ping_host(host):
    os.system(f"ping -c 4 {host}")  # host = "8.8.8.8; rm -rf /"

# VULNERABLE: subprocess with shell=True
import subprocess

def convert_file(filename):
    subprocess.call(f"convert {filename} output.pdf", shell=True)
```

#### Secure Code -- Python

```python
# SECURE: No shell, argument list
import subprocess
import re

def ping_host(host):
    # Validate input
    if not re.match(r'^[a-zA-Z0-9.\-]+$', host):
        raise ValueError("Invalid hostname")
    subprocess.run(["ping", "-c", "4", host], capture_output=True, timeout=30)

def convert_file(filename):
    # Validate filename and use argument list
    if not re.match(r'^[a-zA-Z0-9_.\-]+$', filename):
        raise ValueError("Invalid filename")
    subprocess.run(["convert", filename, "output.pdf"], capture_output=True, timeout=60)
```

#### Secure Code -- TypeScript

```typescript
// SECURE: Use execFile (no shell) instead of exec
import { execFile } from 'child_process';

function pingHost(host: string): Promise<string> {
  // Validate input
  if (!/^[a-zA-Z0-9.\-]+$/.test(host)) {
    throw new Error('Invalid hostname');
  }
  return new Promise((resolve, reject) => {
    execFile('ping', ['-c', '4', host], { timeout: 30000 }, (error, stdout) => {
      if (error) reject(error);
      else resolve(stdout);
    });
  });
}
```

#### Prevention

- [ ] Avoid invoking OS commands from application code; use library APIs instead
- [ ] If shell commands are necessary, use argument arrays (not shell=True)
- [ ] Validate input against strict allowlists
- [ ] Use execFile/execFileSync instead of exec/execSync
- [ ] Escape shell metacharacters if shell invocation is unavoidable

### CWE-94: Code Injection

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 94                                         |
| Name            | Improper Control of Generation of Code (Code Injection) |
| 2025 Rank       | 11                                         |
| 2024 Rank       | 11                                         |
| Change          | Unchanged                                  |
| Affected Languages | Python, JavaScript, PHP, Ruby            |
| CVSS Range      | 8.0 - 10.0 (High to Critical)             |

#### Description

Code Injection occurs when an application dynamically generates and executes code using untrusted input. This includes eval(), exec(), and similar constructs. Attackers can execute arbitrary code in the application's context.

#### Vulnerable Code -- Python

```python
# VULNERABLE: eval() with user input
def calculate(expression):
    return eval(expression)  # expression = "__import__('os').system('rm -rf /')"
```

#### Secure Code -- Python

```python
# SECURE: Use a safe math parser
import ast
import operator

SAFE_OPERATORS = {
    ast.Add: operator.add,
    ast.Sub: operator.sub,
    ast.Mult: operator.mul,
    ast.Div: operator.truediv,
}

def safe_calculate(expression: str) -> float:
    tree = ast.parse(expression, mode='eval')
    return _eval_node(tree.body)

def _eval_node(node):
    if isinstance(node, ast.Constant) and isinstance(node.value, (int, float)):
        return node.value
    elif isinstance(node, ast.BinOp):
        op_func = SAFE_OPERATORS.get(type(node.op))
        if op_func is None:
            raise ValueError(f"Unsupported operator: {type(node.op).__name__}")
        return op_func(_eval_node(node.left), _eval_node(node.right))
    raise ValueError(f"Unsupported expression: {ast.dump(node)}")
```

#### Prevention

- [ ] Never use eval(), exec(), or similar with untrusted input
- [ ] Use purpose-built parsers for mathematical or logical expressions
- [ ] Sandbox any code execution in isolated environments
- [ ] Use static analysis to flag dangerous function calls

```typescript
function searchUser(username: string): Promise<any> {
```

```typescript
import ldapEscape from 'ldap-escape';

function searchUser(username: string): Promise<any> {

}
```

#### Prevention

- [ ] Validate all input against strict allowlists
- [ ] Use parameterized interfaces where available

## Authentication and Authorization Weaknesses

### CWE-862: Missing Authorization

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 862                                        |
| Name            | Missing Authorization                      |
| 2025 Rank       | 8                                          |
| 2024 Rank       | 9                                          |
| Change          | Up 1                                       |
| Affected Languages | All                                      |
| CVSS Range      | 5.0 - 9.8 (Medium to Critical)            |

#### Description

Missing Authorization occurs when an application does not verify that a user is authorized to access a resource or perform an action. The application may check authentication (who you are) but not authorization (what you can do).

#### Vulnerable Code -- TypeScript

```typescript
// VULNERABLE: No authorization check
app.delete('/api/posts/:id', authenticate, async (req, res) => {
  await Post.findByIdAndDelete(req.params.id); // Any user can delete any post
  res.status(204).send();
});
```

#### Secure Code -- TypeScript

```typescript
// SECURE: Authorization enforced
app.delete('/api/posts/:id', authenticate, async (req, res) => {
  const post = await Post.findById(req.params.id);
  if (!post) return res.status(404).json({ error: 'Not found' });
  if (post.authorId !== req.user.id && !req.user.roles.includes('admin')) {
    return res.status(403).json({ error: 'Forbidden' });
  }
  await Post.findByIdAndDelete(req.params.id);
  res.status(204).send();
});
```

#### Prevention

- [ ] Implement authorization checks on every endpoint and action
- [ ] Use a centralized authorization framework
- [ ] Default to deny -- require explicit grants
- [ ] Write authorization tests for all protected resources

### CWE-863: Incorrect Authorization

| Field           | Value                                      |
| --------------- | ------------------------------------------ |
| CWE ID          | 863                                        |
| Name            | Incorrect Authorization                    |
| 2025 Rank       | 12                                         |
| 2024 Rank       | 14                                         |
| Change          | Up 2                                       |
| Affected Languages | All                                      |
| CVSS Range      | 5.0 - 9.0 (Medium to Critical)            |

#### Description

Incorrect Authorization occurs when authorization checks are present but implemented incorrectly, allowing unauthorized access. This includes using user-controlled data for authorization decisions, missing checks on related resources, or incorrect role verification logic.

#### Vulnerable Code -- Python

```python
# VULNERABLE: Client-supplied role used for authorization
@app.route('/api/admin/users', methods=['GET'])
@login_required
def list_users():
    # Trusting client-supplied header
    if request.headers.get('X-User-Role') == 'admin':
        return jsonify(User.query.all())
    return jsonify({"error": "Forbidden"}), 403
```

#### Secure Code -- Python

```python
# SECURE: Server-side role verification
@app.route('/api/admin/users', methods=['GET'])
@login_required
def list_users():
    # Check role from server-side session/database
    if 'admin' not in current_user.roles:
        return jsonify({"error": "Forbidden"}), 403
    users = User.query.all()
    return jsonify([UserSchema().dump(u) for u in users])
```

#### Prevention

- [ ] Never trust client-supplied authorization claims
- [ ] Derive authorization context from server-side session/token
- [ ] Test authorization with different user roles in integration tests
- [ ] Use attribute-based access control (ABAC) or role-based access control (RBAC)

#### Secure Code -- TypeScript

```typescript
// SECURE: Validated file upload
import multer from 'multer';
import crypto from 'crypto';
import path from 'path';

const storage = multer.diskStorage({
  destination: '/uploads',
  filename: (req, file, cb) => {
    // Generate random filename to prevent path traversal
    const ext = path.extname(file.originalname).toLowerCase();

    const ext = path.extname(file.originalname).toLowerCase();
    if (allowed.includes(ext) && allowedMimes.includes(file.mimetype)) {
      cb(null, true);
    } else {
      cb(new Error('Invalid file type'));
    }
  },

  if (!req.file) return res.status(400).json({ error: 'No file' });
```

#### Secure Code -- Go

```go
// SECURE: Atomic transaction with row-level locking
func Transfer(db *sql.DB, fromID, toID string, amount float64) error {
    tx, err := db.BeginTx(context.Background(), &sql.TxOptions{Isolation: sql.LevelSerializable})
    if err != nil {
        return err
    }
    defer tx.Rollback()

    // SELECT FOR UPDATE locks the row
    var balance float64
    err = tx.QueryRow("SELECT balance FROM accounts WHERE id = $1 FOR UPDATE", fromID).Scan(&balance)
    if err != nil {
        return err
    }
    if balance < amount {
        return ErrInsufficientBalance
    }

    _, err = tx.Exec("UPDATE accounts SET balance = balance - $1 WHERE id = $2", amount, fromID)
    if err != nil {
        return err
    }
    _, err = tx.Exec("UPDATE accounts SET balance = balance + $1 WHERE id = $2", amount, toID)
    if err != nil {
        return err
    }
    return tx.Commit()
}
```

## Best Practices by Category

### Memory Safety

1. Use memory-safe languages (Rust, Go, Java, Python, TypeScript) for new projects.
2. For C/C++ codebases, enable compiler protections (stack canaries, ASLR, DEP).
3. Run AddressSanitizer and MemorySanitizer during CI testing.
4. Use static analysis tools (Coverity, CodeQL) to detect memory bugs.
5. Adopt smart pointers and RAII patterns in C++.

### Injection Prevention

1. Use parameterized queries for all database access.
2. Use template engines with auto-escaping for HTML output.
3. Avoid shell command invocation; use library APIs instead.
4. Never use eval() or exec() with untrusted input.
5. Implement Content Security Policy headers.

### Authentication and Authorization

1. Implement authorization checks on every endpoint using centralized middleware.
2. Require authentication for all non-public functions.
3. Use server-side session/token data for authorization decisions.
4. Apply the principle of least privilege to all roles and permissions.
5. Log all authentication and authorization failures.

### Input Validation

1. Define strict input schemas for all endpoints.
2. Validate type, length, format, and range.
3. Use allowlists rather than denylists.
4. Validate file uploads by content, not just extension.
5. Resolve and validate file paths before use.

### Information Protection

1. Return generic error messages to users.
2. Never log passwords, tokens, or credit card numbers.
3. Use response DTOs to control exposed fields.
4. Implement log redaction for sensitive data.
5. Review error handling in code reviews.
