## Core Concepts

### Code Generation Approaches

| Approach | Input | Output | Example Tools |
|----------|-------|--------|---------------|
| Template-based | Data model + template | Boilerplate code | Yeoman, Hygen, Plop |
| Schema-driven | API/DB schema | Typed clients, models | openapi-generator, Prisma, sqlc |
| AST-based | Source code AST | Transformed source | Babel plugins, ts-morph, jscodeshift |
| LLM-based | Natural language | Arbitrary code | Copilot, Claude Code, Cursor |
| Hybrid | Schema + LLM refinement | Context-aware typed code | AI-augmented codegen pipelines |

### When to Use Each Approach

- **Template-based** -- Predictable, repeatable scaffolding (new component, new service)
- **Schema-driven** -- API contracts, database access layers, type definitions
- **AST-based** -- Large-scale refactoring, codemod migrations
- **LLM-based** -- Novel logic, complex business rules, exploratory prototyping
- **Hybrid** -- Schema generates structure, LLM fills implementation details

## Schema-Driven Code Generation

### OpenAPI to SDK Generation

```bash
# Generate TypeScript client from OpenAPI spec
npx @openapitools/openapi-generator-cli generate \
  -i ./api/openapi.yaml \
  -g typescript-axios \
  -o ./src/generated/api-client \
  --additional-properties=supportsES6=true,withInterfaces=true

# Generate Python client
openapi-generator-cli generate \
  -i ./api/openapi.yaml \
  -g python \
  -o ./generated/python-client
```

```yaml
# openapi.yaml - source of truth
openapi: 3.1.0
paths:
  /users/{id}:
    get:
      operationId: getUser
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string, format: uuid }
      responses:
        '200':
          content:
            application/json:
              schema: { $ref: '#/components/schemas/User' }
```

### GraphQL Code Generation

```typescript
// codegen.ts - GraphQL Codegen configuration
import type { CodegenConfig } from '@graphql-codegen/cli';

const config: CodegenConfig = {
  schema: 'http://localhost:4000/graphql',
  documents: ['src/**/*.graphql'],
  generates: {
    './src/generated/graphql.ts': {
      plugins: [
        'typescript',
        'typescript-operations',
        'typescript-react-apollo',
      ],
      config: {
        strictScalars: true,
        enumsAsTypes: true,
        avoidOptionals: { field: true },
      },
    },
  },
};
export default config;
```

```graphql
# src/queries/users.graphql
query GetUser($id: ID!) {
  user(id: $id) {
    id
    email
    profile { displayName avatarUrl }
  }
}
```

### SQL to Type-Safe Code (sqlc)

```sql
-- queries/users.sql
-- name: GetUser :one
SELECT id, email, name, created_at
FROM users
WHERE id = $1;

-- name: ListUsers :many
SELECT id, email, name, created_at
FROM users
ORDER BY created_at DESC
LIMIT $1 OFFSET $2;

-- name: CreateUser :one
INSERT INTO users (email, name)
VALUES ($1, $2)
RETURNING id, email, name, created_at;
```

```yaml
# sqlc.yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "queries/"
    schema: "schema/"
    gen:
      go:
        package: "db"
        out: "internal/db"
        sql_package: "pgx/v5"
        emit_json_tags: true
        emit_interface: true
```

```go
// Generated by sqlc -- internal/db/users.sql.go
type User struct {
    ID        uuid.UUID `json:"id"`
    Email     string    `json:"email"`
    Name      string    `json:"name"`
    CreatedAt time.Time `json:"created_at"`
}

func (q *Queries) GetUser(ctx context.Context, id uuid.UUID) (User, error) {
    row := q.db.QueryRow(ctx, getUser, id)
    var i User
    err := row.Scan(&i.ID, &i.Email, &i.Name, &i.CreatedAt)
    return i, err
}
```

### Prisma Schema to Client

```prisma
// schema.prisma
model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String?
  posts     Post[]
  createdAt DateTime @default(now())
}

model Post {
  id        String   @id @default(uuid())
  title     String
  content   String?
  author    User     @relation(fields: [authorId], references: [id])
  authorId  String
}
```

```typescript
// Generated Prisma client usage -- fully typed
const user = await prisma.user.findUnique({
  where: { id: userId },
  include: { posts: { orderBy: { createdAt: 'desc' }, take: 10 } },
});
// user is typed as User & { posts: Post[] } | null
```

### Protocol Buffers to gRPC Stubs

```protobuf
// user.proto
syntax = "proto3";
package user.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (GetUserResponse);
  rpc ListUsers(ListUsersRequest) returns (ListUsersResponse);
}

message GetUserRequest { string id = 1; }
message GetUserResponse { User user = 1; }
message User {
  string id = 1;
  string email = 2;
  string name = 3;
}
```

```bash
# Generate Go stubs
protoc --go_out=. --go-grpc_out=. proto/user.proto
```

## LLM-Based Code Generation

### Common Failure Modes

| Failure Mode | Example | Detection |
|---|---|---|
| Hallucinated APIs | Calling `fs.readFileAsync()` (does not exist) | Type checking, linter |
| Outdated patterns | Using `componentWillMount` in React 19 | Linter rules, version-aware review |
| Security vulnerabilities | SQL concatenation instead of parameterized queries | SAST scanning |
| Incorrect error handling | Swallowing errors with empty catch blocks | Code review, linter rules |
| Non-existent packages | Importing `left-pad-pro` (does not exist) | Package registry verification |
| Subtle logic errors | Off-by-one in pagination, race conditions | Unit tests, property-based tests |

### Human-in-the-Loop Decision Matrix

| Scenario | Automation Level | Rationale |
|----------|-----------------|-----------|
| Type generation from schema | Fully automated | Deterministic, verifiable output |
| API client generation | Fully automated | Schema is source of truth |
| Boilerplate scaffolding | Fully automated | Template-driven, low risk |
| Business logic implementation | Human review required | Correctness depends on domain knowledge |
| Security-sensitive code | Expert review required | Auth, crypto, access control need expertise |
| Performance-critical code | Human review required | AI often generates naive implementations |
| Database migrations | Human approval required | Data loss risk on incorrect migrations |

### Code Review Checklist for AI-Generated Code

```markdown
## AI Code Review Checklist
- [ ] All imported packages/modules exist and are correctly versioned
- [ ] API calls match actual library signatures (no hallucinated methods)
- [ ] Error handling covers all failure paths
- [ ] No hardcoded secrets, credentials, or sensitive values
- [ ] SQL queries use parameterized statements
- [ ] Edge cases handled (null, empty, boundary values)
- [ ] Code follows project conventions and style guide
- [ ] Tests cover the generated code paths
- [ ] No deprecated APIs or patterns used
- [ ] Performance characteristics are acceptable
```

## Automated Code Generation Pipelines

### CI/CD Integration

```yaml
# .github/workflows/codegen.yml
name: Code Generation
on:
  push:
    paths:
      - 'api/openapi.yaml'
      - 'schema.prisma'
      - 'proto/**/*.proto'
      - 'queries/**/*.sql'

jobs:
  generate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Generate OpenAPI client
        run: |
          npx @openapitools/openapi-generator-cli generate \
            -i api/openapi.yaml -g typescript-axios \
            -o src/generated/api-client

      - name: Generate Prisma client
        run: npx prisma generate

      - name: Generate sqlc
        run: sqlc generate

      - name: Generate protobuf stubs
        run: buf generate

      - name: Check for uncommitted changes
        run: |
          git diff --exit-code src/generated/ || \
            (echo "Generated code is out of date. Run codegen locally." && exit 1)
```

### Schema-First Development Workflow

```
1. Define schema (OpenAPI / GraphQL / Protobuf / SQL)
        ↓
2. Run code generation (CI or pre-commit hook)
        ↓
3. Generated types/clients available to developers
        ↓
4. Implement business logic using generated interfaces
        ↓
5. Schema change → re-generate → type errors reveal breakage
```

## 10 Best Practices

1. **Treat schemas as the single source of truth.** Generate code from schemas, never maintain generated code manually. Schema changes propagate automatically to all consumers.

2. **Commit generated code to version control.** Make diffs visible in code review. Reviewers catch unintended schema changes through generated code diffs.

3. **Run code generation in CI to detect drift.** Fail the build when generated code does not match what the schema produces. Prevent stale generated code from reaching production.

4. **Use type-safe code generation over string-based templates.** Prefer tools like sqlc, Prisma, and GraphQL Codegen that produce fully typed output. Catch errors at compile time, not runtime.

5. **Validate AI-generated code with automated checks.** Run type checker, linter, SAST scanner, and tests on all LLM-generated code before accepting it.

6. **Separate generated code from handwritten code.** Place generated code in `generated/` or `__generated__/` directories. Never edit generated files directly.

7. **Version your code generation tooling.** Pin openapi-generator, protoc, sqlc, and Prisma versions. Inconsistent generator versions cause unnecessary diffs.

8. **Add generation commands to package.json or Makefile.** Standardize how developers run code generation: `npm run generate` or `make codegen`.

9. **Use LLMs to augment, not replace, schema-driven generation.** Let deterministic tools generate the structure. Use LLMs to fill in implementation details, documentation, and test cases.

10. **Review AI-generated code for hallucinated APIs.** Verify that every function call, import, and package reference actually exists. Cross-check against official documentation.

## Anti-Patterns

| Anti-Pattern | Impact | Fix |
|---|---|---|
| Manually editing generated files | Changes overwritten on next generation | Extend via partial classes, wrappers, or config |
| No CI validation of generated code | Stale types cause runtime errors | Add `git diff --exit-code` check in CI |
| Using LLMs for deterministic codegen tasks | Non-reproducible output, inconsistent types | Use schema-driven generators for typed APIs |
| Generating without schema validation | Invalid schemas produce broken code | Validate schemas before running generators |
| Skipping code review for generated code | Unnoticed schema changes, security issues | Review generated diffs like any other code |
| Coupling business logic to generated types | Painful migrations when schema changes | Use adapter/mapper layer between generated and domain types |
| Not pinning generator versions | Different developers get different output | Lock versions in package.json or go.mod |
| Generating entire applications with LLMs | Unmaintainable, inconsistent codebase | Use LLMs for discrete tasks, humans for architecture |

## Enforcement Checklist

- [ ] Schema files (OpenAPI, GraphQL, Protobuf, SQL) are the source of truth for generated code
- [ ] Code generation runs in CI and fails on drift
- [ ] Generated files are in dedicated directories and clearly marked
- [ ] Generator tool versions are pinned and consistent across team
- [ ] `npm run generate` / `make codegen` command exists and is documented
- [ ] AI-generated code passes type checking, linting, and SAST before merge
- [ ] Code review process includes checklist for AI-generated code
- [ ] Generated code is committed to version control with visible diffs
- [ ] Pre-commit hooks regenerate code when schema files change
- [ ] Team understands when to use schema-driven vs LLM-based generation
